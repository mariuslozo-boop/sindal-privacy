Sindal Chrome Extension, Privacy Policy

Last updated: 2026-05-13

Who we are

The Sindal Chrome extension is an internal tool for staff of Sindal Autoophug (a Danish auto-recycler). It coordinates a parts-lookup workflow across three third-party websites the staff already use, plus an optional browser-based Lager (stock) lookup pane.

Contact: mariuslozovskis@gmail.com

What the extension does

The extension assists with one workflow: moving a vehicle identifier (VIN) and an original-equipment part number (OE) between three trade websites (au2web.dk, partslink24.com, appmain.nemdele.dk) and, optionally, the user's own stock spreadsheet via a browser-native Lager pane. It is not intended for the general public.

Data we collect

1. Workflow telemetry

When the extension completes a workflow step or performs a Partslink24 keep-alive reload, it sends a single event to a Google Apps Script endpoint owned by the extension author. Each event contains only:

  • the step name, one of: au2web_capture, partslink24_vin_fill, nemdele_send (each triggered by a user action), or partslink24_keepalive_reload (triggered internally by the service worker when the user has been away from Partslink24 for 5 minutes),
  • a short random flow-correlation id used to group the steps of one lookup together,
  • a tester identity the user can pick from a dropdown in the popup (a fixed list of "bruger-1" through "bruger-10"); if the user has not picked one, the literal value "unknown" is sent in its place,
  • the local timestamp.

No VIN, OE number, vehicle data, page content, IP address, browsing history, or other identifier is sent. Telemetry is always active while the extension is installed; it cannot currently be turned off from inside the extension. The endpoint receives the request from the user's browser, which means the receiving Google service necessarily sees the user's network-level IP address; the extension itself does not collect or store it.

2. Local-only data

The extension stores the following in chrome.storage on the user's own computer. This data never leaves the device:

  • the selected tester identity,
  • the last opened popup pane (main, Lager, or Indstillinger),
  • the last selected match-report filename,
  • the current vehicle context (VIN, plate, last OE number), cleared automatically on each Chrome restart,
  • the keep-alive, auto-switch, and debug-mode toggle states,
  • the last keep-alive outcome (a short status string and timestamp, shown to the user only when Debug-tilstand is enabled),
  • a transient brand-pick state used by the Partslink24 content script,
  • activity timestamps used by the Partslink24 keep-alive (cleared on browser restart).

3. Lager pane (browser-native stock lookup)

The Lager pane lets the user select a local stock spreadsheet (stock.xlsx) and drag in parts-rapport PDFs to see which parts are in stock. All parsing happens inside the browser using bundled SheetJS and PDF.js libraries; parsed stock rows are cached in the browser's IndexedDB so subsequent rapport checks are instant. Nothing from the stock file or the rapport PDFs is uploaded, and no separate native program is installed or launched.

How data is used

Telemetry is used solely to measure whether the workflow is being completed end-to-end. It is not sold, shared with third parties, used for advertising, used for profiling, or used to determine creditworthiness or lending eligibility.

Where data is stored

Telemetry events are written to a Google Sheet owned by the extension author and accessed only by Sindal staff for internal review. Events are retained indefinitely and can be deleted on request to the contact email above.

What we do NOT collect

  • Personal identifiers beyond the user-chosen display name
  • VINs, license plates, OE numbers, or any parts data
  • Stock spreadsheet contents or rapport PDF contents
  • Page content from au2web, Partslink24, or NemDele
  • Authentication credentials or session cookies
  • Geolocation
  • Browsing history outside the three host-permission domains
  • Health or financial information

Permissions and why we need them

  • storage, saves the local preferences listed above via chrome.storage.local and chrome.storage.session. The Lager pane's stock cache uses IndexedDB, which is available to extensions without an additional permission.
  • tabGroups, coordinates the workflow tabs in a single Chrome group named "Sindal".
  • alarms, schedules a one-shot reload of the user's Partslink24 tab 5 minutes after the user focuses away from Partslink24, so the session does not expire while the user is working elsewhere. The alarm is cancelled the moment the user focuses Partslink24 again.
  • Host permission on *.partslink24.com, webkat.au2web.dk, appmain.nemdele.dk, required to inject the workflow content scripts.
  • Host permission on script.google.com and script.googleusercontent.com, required for the telemetry endpoint.
